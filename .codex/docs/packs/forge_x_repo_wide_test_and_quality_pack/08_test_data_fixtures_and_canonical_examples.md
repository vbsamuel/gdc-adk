# Test Data Fixtures and Canonical Examples

## Purpose

Test data must be canonical enough that repeated implementation sessions converge on the same behavior.

## Required fixture classes
- canonical request payloads
- canonical normalized artifacts
- canonical issues and findings
- canonical workflow runs
- canonical continuity snapshots
- canonical provider success and failure responses
- canonical multi-agent handoff artifacts

## Fixture rules
- fixtures must be named semantically
- fixtures must use controlled taxonomy values
- fixtures must not hide optional fields that matter to behavior
- fixtures must represent both happy and unhappy paths
